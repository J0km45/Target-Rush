* **장르**: `FPS 슈팅 게임`
* **플레이타임**: 약 2분
* **개발 환경**: `Unity 6000.3.9f1`
* **플랫폼**: `Windows`
* **프로젝트 기간**: 2026.02.26 ~ 2026.03.06 (1주일)

## ✨ 게임 특징 (Features)

제한 시간 안에 맵을 돌아다니며, 등장하는 타겟을 맞춰 점수를 획득하는 방식으로 진행된다.

---

### 핵심 메커니즘 1: [타겟 사격]

- 플레이어가 조준과 사격을 통해 타겟을 맞추는 방식

### 핵심 메커니즘 2: [랜덤 타겟 스폰 시스템]

- 맵의 여러 위치에서 타겟이 랜덤하게 생성되어, 플레이어가 탐색하며 발견해야함

### 핵심 메커니즘 3: [제한 시간 점수 시스템]

- 제한 시간 내에 타겟을 맞춰 점수를 획득하는 타임 어택 방식
- 최고 점수를 갱신하면 기록을 저장

## 🛠️ 구현 기능
### 1. 게임 진행 시스템

**1-1.** **구현 목표**

- 제한 시간과 점수에 따라 게임 상태가 실시간으로 변경되는 상황에서, 게임 시작부터 일시정지와 게임 종료까지의 전체 흐름을 일관되게 관리해야 했다.

**1-2. 구현 방법**

- **MVP 패턴 적용**
    - Model은 게임 상태를 관리하고, View는 화면 표시만 담당하도록 구성했고, Presenter가 Model과 View 사이의 상호작용을 중재하도록 하여 각 클래스가 하나의 역할에 집중하도록 구현했다.
- **최고 점수 저장**
    - `PlayerPrefs`를 이용해 최고 점수를 저장하고, 게임 종료 시 현재 점수와 비교하여 최고 점수를 갱신하도록 구현했다.

**1-3. 설계 의도**

- 게임 로직이 View의 구체적인 UI 구성에 직접 의존하지 않도록 하여, UI를 수정하거나 새로운 화면을 추가할 때 게임 로직의 변경을 최소화하고자 했다.
- 최고 점수는 구조가 단순하고 저장할 데이터의 양이 적기 때문에 별도의 파일이나 서버 대신 `PlayerPrefs`를 사용했다.

**1-4. 결과**

- 게임 진행과 UI를 분리하여 기능 수정 및 유지보수가 용이한 구조를 구성할 수 있었다.
- 최고 점수를 로컬에 저장하여 게임을 재실행해도 이전 기록을 유지할 수 있었다.

**1-5. 회고**

- MVP 패턴을 적용하여 게임 로직과 UI의 역할을 분리할 수 있었다. 다만 일부 게임 상태를 Update()에서 매 프레임 확인하도록 구현했는데, 상태가 변경되는 시점에 이벤트를 발생시키는 방식으로 구성했다면 불필요한 반복 처리를 줄이고 상태 변경 흐름을 더욱 명확하게 관리할 수 있었을 것 같다.
- 현재는 `PlayerPrefs`를 이용해 최고 점수를 로컬에 저장했다. 추후에는 Firebase 같은 서버 기반 저장 방식을 적용하여 최고 점수를 관리하고, 실시간 랭킹 기능까지 확장해보고 싶다.

<br>

### 2. 플레이어 및 총기 시스템

**2-1.** **구현 목표**

- 플레이어의 이동과 총기 조작이 동시에 발생하는 상황에서, 조준과 재장전 상태에 따라 이동 속도와 입력을 적절히 제한하며 일관된 조작감을 제공해야 했다.

**2-2. 구현 방법**

- **Input System 및 CharacterController 기반 플레이어 이동**
    - `Unity Input System`으로 플레이어 입력을 처리하고, `CharacterController`를 이용해 이동과 중력을 구현했다.
    - 조준 및 재장전 상태에 따라 이동 속도와 입력을 제어하여 자연스러운 조작이 가능하도록 구성했다.
- **Raycast 기반 사격 판정**
    - 카메라 중심에서 `Raycast`를 발사하여 조준한 대상의 피격 여부를 판정했다.
    - 충돌한 대상이 `IHittable` 인터페이스를 구현한 경우 피해 처리를 호출하여, 다양한 대상에 동일한 방식으로 적용할 수 있도록 구현했다.

**2-3. 설계 의도**

- 입력 처리를 `Unity Input System`으로 구성하여 키보드와 마우스 입력을 한곳에서 관리하고, 추후 입력 방식이 변경되더라도 이동 및 사격 로직의 수정을 최소화하고자 했다.
- 플레이어 이동에는 `CharacterController`를 사용하여 Rigidbody 기반 물리 이동보다 직접적이고 안정적으로 이동, 충돌, 중력을 제어하고자 했다.
- 총알 오브젝트를 실제로 생성하는 대신 `Raycast`로 사격을 판정하여, 즉시 명중하는 총기의 특성을 표현하고 불필요한 오브젝트 생성과 물리 연산을 줄이고자 했다.
- `IHittable` 인터페이스를 통해 피격 처리를 공통화하여, 새로운 피격 대상을 유연하게 추가할 수 있도록 했다.

**2-4. 결과**

- `CharacterController`를 활용하여 물리 연산의 영향을 최소화하면서 안정적인 플레이어 이동을 구현했다.
- `Raycast` 기반 사격 판정을 통해 즉각적인 명중 처리를 구현하고, 불필요한 오브젝트 생성 없이 효율적으로 사격을 처리했다.
- `IHittable` 인터페이스를 적용하여 새로운 피격 대상도 쉽게 확장할 수 있는 구조를 구현했다.

**2-5. 회고**

- MVP 패턴을 적용하여 총기 상태를 `Model`, UI 표시를 `View`로 분리했지만, Presenter 역할을 담당하는 `GunController`에 여러 기능의 책임이 다소 집중되었다. 추후에는 입력, 무기 동작, 연출 처리를 별도의 컴포넌트로 분리하여 각 클래스의 역할을 더욱 명확하고 유지보수하기 쉬운 구조로 개선해보고 싶다.

<br>

### 3. 타겟 시스템

**3-1.** **구현 목표**

- 플레이어가 타겟을 제거할 때마다 점수를 반영하고 새로운 타겟을 생성하면서, 동일한 위치에 타겟이 중복 생성되지 않도록 관리해야 했다.

**3-2. 구현 방법**

- **스폰 위치 관리**
    - 타겟 생성 위치를 `List<Transform>`으로 관리하고, 무작위 위치를 선택하여 타겟을 생성했다.
    - 생성에 사용된 위치는 목록에서 제거하고, 타겟이 제거되면 다시 추가하여 동일한 위치에 중복 생성되지 않도록 구현했다.
- **이벤트 기반 타겟 관리**
    - Target이 피격되면 `OnDestroyed` 이벤트를 호출하고, `TargetSpawner`에서 이를 구독하여 점수 반영과 타겟 재생성을 처리하도록 구현했다.
    - 타겟이 제거될 때마다 새로운 타겟을 생성하여 최대 타겟 수를 유지하도록 구성했다.

**3-3. 설계 의도**

- 스폰 위치를 리스트로 관리하여 이미 사용 중인 위치를 쉽게 제외하고, 타겟이 중복 생성되지 않도록 했다.
- 타겟 제거는 `OnDestroyed` 이벤트를 통해 알리도록 구성하여, 타겟과 점수 및 스폰 관리 로직의 결합도를 낮추고 각 객체의 역할을 분리하고자 했다.

**3-4. 결과**

- 동일한 위치에 타겟이 중복 생성되지 않도록 구현하여 안정적으로 타겟을 관리할 수 있었다.
- 타겟 제거 시 점수 반영과 새로운 타겟 생성이 자동으로 이루어져 게임 진행이 자연스럽게 이어질 수 있었다.
- 이벤트 기반으로 타겟 제거를 처리하여 점수 관리와 타겟 생성 로직을 독립적으로 유지할 수 있었다.

**3-5. 회고**

- 현재는 타겟 제거 시 `Destroy` 후 새로운 타겟을 `Instantiate`하는 방식으로 처리했다. 추후에는 최대 타겟 수만큼 오브젝트를 미리 생성하고, 피격된 타겟을 비활성화한 뒤 새로운 스폰 위치로 이동시켜 재사용함으로써 반복적인 오브젝트 생성과 삭제에 따른 비용을 줄여보고 싶다.
